# レビューのルール

- 命名の一貫性（スペル、用語、クラス名/ファイル名の統一）
- 型契約の整合性（型注釈と実際の受け渡し・実装が一致しているか）
- ドメイン責務の明確さ（Entity / VO / DomainService の役割分離）
- レイヤ分離と依存方向（domain -> application -> infra の崩れがないか）
- 例外設計（例外型の具体性、原因が追いやすいか）
- 副作用設計（None 返却・in-place更新の意図が読み取れるか）
- 可読性（追いやすさ、重複、ノイズの少なさ）
- 複雑さ（分岐・状態遷移の理解コスト）
- テスト品質（観点網羅、アサーションの強さ、回帰検知力）
- 保守性・拡張性（仕様変更時の影響範囲と直しやすさ）
